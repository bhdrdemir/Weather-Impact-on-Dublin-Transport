Step 1 — Data Acquisition

Pulls data from 3 free public APIs with no keys needed. Open-Meteo gives us hourly weather for Dublin from 2022 to now. CSO PxStat gives monthly bus and Luas passenger numbers. Irish Rail gives live train positions.

resp = requests.get('https://archive-api.open-meteo.com/v1/archive', params={'latitude': 53.35, 'longitude': -6.26, 'hourly': 'temperature_2m,rain,wind_speed_10m,...'}) weather_df = pd.DataFrame(resp.json()['hourly'])

Step 2 — Feature Extraction

Creates new columns from the raw data. Temporal features like is_weekend, is_rush_hour. Weather features like rain_category, weather_severity (composite 0-10 index). Then rolls up hourly data into daily and monthly aggregates.

def compute_weather_severity(temp_c, rain_mm, wind_kmh): cold = max(0, (10 - temp_c) / 20) # 25% weight rain = min(rain_mm / 30.0, 1.0) # 50% weight wind = min(wind_kmh / 60.0, 1.0) # 25% weight return round(cold*2.5 + rain*5.0 + wind*2.5, 2)

Step 3 — Transformations & Merging

Merges monthly weather with monthly transport on (year, month). Drops COVID years 2020-2021 so the correlation is not dominated by lockdowns. Creates interaction features like bus_share and weather_impact_score.

1. Project Overview

This project implements a full Data Acquisition & Preprocessing Pipeline for analysing how weather affects public transport ridership in Dublin. The deliverable is a local Flask web application that fetches live data from four public APIs, runs a feature-extraction pipeline, stores the output in SQLite, and serves an interactive dashboard with real-time weather-based transport predictions.

Assignment scope per the B9AI001 CA2 brief: API-based data acquisition, feature extraction, transformation, database loading, unit + integration testing, and documentation — all tied together by a reproducible end-to-end pipeline. The dashboard also applies historical correlation coefficients to live weather data to produce delay risk and crowdedness predictions for Dublin Bus, Luas, and Irish Rail.

2. System Architecture

User's Browser (index.html, Chart.js) | | HTTP (auto-refresh 60s, client-side cache) v Flask Backend (app.py) | +-- GET /api/live -> ALL 9 areas weather + Luas/Rail (single call) +-- GET /api/bus/arrivals -> estimated bus arrivals with delay/crowd predictions +-- GET /api/prediction -> delay risk from live weather + DB correlations +-- POST /api/pipeline/run -> background thread runs full pipeline +-- GET /api/features/sample -> sample rows from every DB table +-- GET /api/dashboard/data, /api/stats -> historical analytics +-- GET /api/tables, /api/query -> DB browser + SQL editor +-- GET /api/tests/run -> unit + integration tests | v Pipeline (pipeline.py) <-> SQLite (dublin_transport.db, 8 tables) | v External APIs: - Open-Meteo (historical archive + live multi-location forecast) - CSO PxStat REST (monthly bus + Luas passengers, JSON-stat 2.0) - Irish Rail (real-time train positions, XML) - Luas Forecasting (real-time tram arrivals per stop, XML)

The /api/live endpoint fetches all 9 Dublin districts in a single Open-Meteo multi-location call, then fetches Luas and Irish Rail in parallel using ThreadPoolExecutor. All data is cached client-side so area switching is instant with no reload.

3. Data Acquisition

Four free public APIs. No API key required for any pipeline step.

Open-Meteo Historical Archive

Hourly weather variables for Dublin. Single call fetches 2022-01-01 to 2024-12-31.

6. Database Schema

Eight normalised tables in SQLite:

  • weather_hourly — raw hourly weather + temporal + weather features (~26k rows)
  • weather_daily — daily aggregates (~1.1k rows)
  • weather_monthly — monthly aggregates (~36 rows)
  • bus_passengers — Dublin Bus monthly + transport features
  • luas_passengers — Luas monthly + transport features
  • weather_transport_merged — merged + interaction features (used for correlation)
  • rail_realtime — snapshot of active Irish Rail trains at pipeline run-time
  • luas_realtime — reserved; tram arrivals served live from /api/live instead

8. Statistical Analysis

  • Pearson correlation — linear relationship between weather variables and the delay proxy
  • Spearman correlation — rank-based, catches monotonic (non-linear) trends
  • Kruskal-Wallis H-test — whether delay-proxy medians differ significantly across weather groups (rain_group, temp_group, severity_group)

All computed on-the-fly from the merged DB table, served via /api/stats.

9. Live Prediction Model

The dashboard uses the historical correlations to predict delay risk and crowdedness in real time:

  • Delay risk — for each live weather reading (temperature, rain, wind), the system scores risk using the Pearson correlation coefficients from the historical DB. A positive correlation between rainfall and delay proxy means more rain → higher delay risk. The score is mapped to a 0–4 level (Normal → Severe disruption).
  • Crowdedness — predicted using Pearson correlation coefficients between weather variables (rainfall, temperature, wind speed, severity) and historical passenger counts from the ML pipeline. Rush-hour and late-night adjustments are layered on top. The same weather produces the same crowdedness at a given stop — no random variation.
  • Bus arrival estimates — base frequencies adjusted by weather-driven delay factors. Each route has rush-hour and off-peak intervals; weather severity adds proportional delay.
severity = compute_weather_severity(temp, rain, wind) # 0-10 weather_delay_pct = severity * 4.0 # up to 40% longer wait adjusted_freq = base_freq * (1.0 + weather_delay_pct / 100) # Crowdedness: correlation-weighted score from historical passenger data r_rain = df['total_rain'].corr(df['bus_passengers']) score += r_rain * rain_pct * 1.5 # repeat for wind, temp, severity
